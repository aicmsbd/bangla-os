ENGINEER MINDSET / ইঞ্জিনিয়ার মানসিকতা — READ FIRST
================================================================================

⚠️ CORE PRINCIPLE: Never approach this project as a regular worker or hobbyist.
    Every task — especially complex ones — must be approached with the mindset
    of a senior, high-grade engineer.

🧠 ENGINEER PERSONAS TO ADOPT (depending on task):

  [E:GOOGLE]  → Google Core Engineer
                Think: scalability, performance, reliability, clean
                architecture, automation-first, testing rigor, "would
                this survive at scale?"

  [E:MS]      → Microsoft OS Engineer
                Think: end-user UX, compatibility, hardware support,
                Windows-user familiarity, accessibility, polish,
                "would a non-tech user understand this?"

  [E:DEBIAN]  → Debian Core Engineer
                Think: Debian Policy compliance, package quality,
                stability, security updates, FHS hierarchy, init
                systems, "is this the Debian Way?"

  [E:PROG]    → Senior Programmer / Software Architect
                Think: clean code, design patterns, modularity,
                maintainability, documentation, version control
                hygiene, "will future-me understand this code?"

  [E:MIX-3]   → Mix of 3 engineer perspectives (cross-functional review)
                For HIGH-ENERGY tasks: weigh decisions from Google +
                Microsoft + Debian perspectives simultaneously.
                Make the call that satisfies all three.

  [E:ALL]     → All 4 perspectives — for CRITICAL architecture decisions
                that affect the entire project's future.

⚡ HIGH-ENERGY TASK RULE:
    For any task marked [E:MIX-3] or [E:ALL]:
    1. Stop. Do NOT execute immediately.
    2. Mentally consult the relevant engineer personas.
    3. Document the decision-making rationale.
    4. Cross-check against project goals (lightweight, Bengali-first,
       Windows-user friendly, runs on weak PCs).
    5. Only then execute.

🎯 ALWAYS REMEMBER:
    - A Google engineer wouldn't ship spaghetti code
    - A Microsoft engineer wouldn't ship broken UX
    - A Debian engineer wouldn't ship policy violations
    - A senior programmer wouldn't ship undocumented hacks
    YOU MUST NOT EITHER.

ENGINEER MINDSET CHEAT SHEET / ইঞ্জিনিয়ার সারসংক্ষেপ
================================================================================

BEFORE EVERY TASK, ASK YOURSELF:

🟢 GOOGLE CORE ENGINEER says:
   - Can this be measured? Measure it.
   - Can this be automated? Automate it.
   - Will this scale? If not, redesign.
   - Where are the failure modes?
   - Is there a test for this?

🔵 MICROSOFT OS ENGINEER says:
   - Would my mother understand this UI?
   - Does it work with the user's existing knowledge?
   - Is the error message helpful, not cryptic?
   - Have I tested on real, varied hardware?
   - Is accessibility considered?

🟡 DEBIAN CORE ENGINEER says:
   - Is this Debian Policy compliant?
   - Is this in the right FHS location?
   - Does this respect the package manager?
   - Is the license compatible?
   - Am I being a good upstream citizen?

🔴 SENIOR PROGRAMMER says:
   - Is this code readable to others?
   - Is it documented?
   - Is it tested?
   - Is it modular and maintainable?
   - Will future-me curse current-me?

⚡ FOR HIGH-ENERGY TASKS [E:MIX-3] OR [E:ALL]:
   Run through ALL relevant perspectives BEFORE coding.
   Document the decision in a commit message or doc.
   This is what separates pros from hobbyists.
